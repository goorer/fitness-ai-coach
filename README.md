# Fitness AI Coach

トレーニング記録とAIによる分析・提案を行うフィットネスアプリ

## Design Philosophy

このアプリは、筋トレの記録を「残すだけ」で終わらせず、次の行動に変えるために作っています。

筋トレは、やった内容を忘れやすい一方で、少しずつ積み上げるほど意味が出ます。重量、回数、種目、日付のような地味なデータでも、続けて記録すれば「前回より伸びたのか」「今は停滞しているのか」「次に何をすればいいのか」を判断する材料になります。

このアプリでは、まず正しく記録できることを優先します。そのうえで、履歴やボリュームを見て、AIが次回のトレーニングの考え方を補助する形を目指します。AIに全部を任せるのではなく、ユーザーが自分の状態を見直すための補助線として使う方針です。

今はコードを書くこと自体のハードルが下がっています。そのため、このプロジェクトでは「作れること」よりも、「何を記録し、何を判断材料にし、どこまでをAIに任せるか」を重視しています。筋トレ記録アプリとしての価値は、画面やAPIの数ではなく、継続して記録したくなることと、記録したデータが次の行動に変わることにあると考えています。

## Considerations

このプロジェクトでは、最初から大きな機能を作り込まず、Exercise、Workout、WorkoutSetのように基本的なデータ構造から順番に組み立てています。記録アプリとしての土台が弱いと、あとからAI分析を足しても説得力がなくなるためです。

APIは、学習しながら拡張しやすいように、まずはFastAPIとSQLiteでシンプルに作っています。将来的にユーザー管理やクラウドDBが必要になる可能性はありますが、現段階では個人利用と検証速度を優先しています。

AI分析では、OllamaをDocker Composeで立ててローカルで動かす構成にしています。外部APIに依存しないため、試行錯誤しやすく、トレーニング記録のような個人的なデータも扱いやすいです。一方で、ローカルLLMは出力が揺れるため、レスポンスを自由文だけにせず、summary、recommendation、cautionsのような構造で返すようにしています。

このアプリは、単なる記録ツールよりもAI分析を中心機能にする前提で設計しています。そのため、LLM呼び出しは直接HTTPで固定せず、LangChainを挟む方針にしています。今はOllamaを使っていますが、将来的に別モデルへ差し替える場合や、過去の記録、種目情報、フォームの注意点などをRAGで参照する場合に、アプリ本体のAPI設計を大きく崩さず試せるようにするためです。

LLMに渡すプロンプトは、指示部分を英語、出力指定を日本語にしています。`llama3.2` で簡単に比較したところ、同じ意味の短い指示では英語の方が少ないトークン数で扱えました。ただし、このアプリで必要なのは日本語の分析結果なので、モデルへの制約やJSON形式の指示は英語で書き、ユーザーに返す文章は日本語に固定しています。

また、AIの提案はあくまで参考情報として扱います。筋トレは体調、フォーム、疲労、睡眠など、記録だけでは判断できない要素が多いため、無理な増量や医療的な判断をしないようにプロンプト側でも制限しています。

設計としては、アプリが断定しすぎないことも意識しています。ユーザーにとって必要なのは、命令ではなく判断材料です。次回重量の候補、注意点、直近の傾向を出しつつ、最終的な判断はユーザー側に残す形にしています。そのため、まずは小さく動くものを作り、実際に記録を入れながら改善していく方針です。

## Features (v1)
- トレーニング記録
- 重量・回数の履歴管理
- グラフによる成長可視化
- AIによる次回トレーニング提案

## Tech Stack
- Frontend: Next.js
- Backend: FastAPI
- DB: SQLite
- AI: Ollama, LangChain
